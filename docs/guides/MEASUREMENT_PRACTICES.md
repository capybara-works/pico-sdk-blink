# 計測の作法 (Measurement Practices)

実機の計測値で原因を切り分ける際の手順集(Living)。判定・断定のルールは
[../operations/AGENT_OPERATION.md](../operations/AGENT_OPERATION.md) の
「計測・診断結果の解釈ディシプリン」を参照。

## ADC計測(RP2040)

RP2040のADCは12bit(0–4095)、基準は3.3V。**高インピーダンス源(分圧器・内蔵センサ等)を
読むときは、サンプル&ホールドの整定が必要**。これを怠ると値が過小評価され、しばしば
「ハード不良」と誤認される。

正しい手順:

1. `adc_init()`
2. 対象がGPIOなら `adc_gpio_init(<gpio>)`
3. `adc_select_input(<input>)`(muxを固定)
4. 数ms待つ(`sleep_ms(2)` 程度)
5. 最初の十数サンプルを**捨てる**(mux切替直後は不正確)
6. 64〜256回**平均**する
7. 換算する: `adc_mv = raw * 3300 / 4095`、分圧があれば物理値 `= adc_mv * 分圧比`
8. 期待値と照合する(例: USB給電中のVSYSはおおむね4.5〜5.2V)

実装例は `blink.cpp` の `adc_avg()` / `run_post()`。

### やってはいけない

- **ADC分圧/センサ入力ピンに pull-up / pull-down 診断をかけない。** 分圧ノードの電圧が
  変わるうえ、過渡やS/H残留電荷でpull-downなのに値が上がる等の**アーティファクト**が出て、
  「フローティング」と誤読する原因になる。
- **1回読みで結論しない。** 高Z源は1回読みで低く出る。
- **生値と物理値を混同しない。** ログ名で `*_raw` / `*_adc_mv` / 物理推定値 を分ける。

### 単位・名前の付け方(必須)

多義的な名前(例 `vsys_mv` が「VSYS本体」か「ADC入力電圧」か不明)は事故を誘発する。
層ごとに別名で出す:

```text
gp29_raw=2058       # ADC生カウント(0-4095)
gp29_adc_mv=1659    # ADCピンの電圧(mV)
vsys_est_mv=4975    # VSYS推定(=gp29_adc_mv × 3)
```

## ワークドエグザンプル: GP29 / VSYS(2026-06-16)

- **症状**: POSTの `vsys` が ~1.7V と低く出た。
- **誤った結論(撤回)**: 「GP29にVSYS/3分圧が無い/RP2040クローン」。
- **真因**: 捨て読み1回のみで、~67kΩの高インピーダンス分圧に対しADCのS/Hが整定せず過小評価。
  同じ診断ログの `nopull raw=2104`(≒5.1V)が真値を示していたのに見落とした。
- **正しい測定**: 整定+16捨て読み+256平均 → `gp29_raw=2058 vsys_est_mv=4975`(=4.98V, 正常)。
- **教訓**: 反証データ(`2104`)を先に説明する。測定手順を疑う。ADC分圧ピンにpull試験をかけない。
  記録(docs/memory)へは確認測定後にのみ断定形で書く。

## 物理ピンとGPIO番号

物理ピン番号とGPIO番号は別物(例: 物理ピン29はGP22であり、GP29ではない)。
GP29はヘッダに露出しない内蔵VSYS/3センス。ピン表記は常にどちらかを明示する。
