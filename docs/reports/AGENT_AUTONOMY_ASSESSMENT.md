# AIエージェント自律性アセスメント

- **日付**: 2026-06-15
- **問い**: この環境で「AIはどこまで自律的に作業でき、どこが依然としてネックか」
- **区分**: Record(実施時点のスナップショット。後から書き換えない)
- **根拠**: `scripts/` 全入口(17本)、`scripts/summarize_evidence.py` の証拠スキーマ(9種)、
  `.github/workflows/ci.yml`、`docs/design/MCP_SETUP.md`、ホストのツール在不在、
  および Phase 2 OLED/ロジアナ実機ブリングアップ([PHASE2_OLED_LOGIC_BRINGUP_REPORT.md](PHASE2_OLED_LOGIC_BRINGUP_REPORT.md))で
  実際に発生した人間との往復。

## 1. 自律性マップ(ループの層ごと)

検証ループ「ビルド→シミュレーション→フラッシュ→観測→判定」を層に分解して評価する。

| 層 | AIの自律度 | 根拠 |
|---|---|---|
| ビルド (CMake/GCC) | 完全自律(CIでも) | `build_result.json`、ヘッドレス |
| 単体テスト (ctest) | 完全自律 | `ctest_result.json` |
| シミュレーション (Wokwi) | 完全自律。**仮想OLEDのピクセルも `wokwi-cli --screenshot-part` で取得可** | wokwi pass + スクショ実証 |
| 証拠からの判定 | 完全自律 | `*_result.json` / log → `verification.md` |
| 環境の自己改善 | 完全自律(probe追加・既定修正・知見記録を実証) | 本セッションのコミット群 |
| フラッシュ (openocd + Debug Probe) | ゲート + 物理probe接続が前提 | `PICO_HARDWARE=1` |
| デジタル観測 (UART/ロジアナ/GDB) | ゲート + **人間が物理配線**してから | `PICO_LOGIC_*` |
| 実機OLEDの実ピクセル | **不可能**(カメラなし = 人間が目) | 本セッション最大の往復要因 |
| Wokwiキャンバスの幾何 | AIから見えない(座標取得手段なし) | waypoint手探りの原因 |

**要点: ループの左半分(ソフト/シム/判定/自己改善)はほぼ完全自律。
右半分(物理アクチュエーション + 物理観測)で人間に依存する。**

## 2. 人間依存は2種類あり、性質が異なる

### (A) 意図的ゲート(設計であり欠陥ではない)
`PICO_HARDWARE` / `PICO_LOGIC_*`。安全のための意図的な摩擦で、AIが正しく止まる。
コストは「安全」ではなく**スループット(人間承認の往復回数)**。
本セッションではフラッシュ承認・ロジアナ承認などで複数回発生したが、これは仕様どおり。

### (B) 物理センシングの欠落(能力の天井 = 本質的ネック)
AIは次を直接観測できない: **実機OLEDの表示・配線の挿さり具合・VCCに電気が来ているか・
Wokwiキャンバス・実機LED/基板の状態。** 人間がセンサーになる。
本セッションで費やした時間の大半(「文字は出ているか」「どのピンに挿したか」「挿し直したか」)は
すべてここに帰着する。**これがこのLabの真の天井。**

## 3. 本セッションで改善した点(実測ベース)

- **sigrok-cli 導入**でロジアナが stub → 実測に昇格(`logic_uart`/`logic_i2c` pass実績)。
- **`scripts/probe_logic_activity.sh`** でデジタル切り分けが再現可能に(「SCLタップ未接続」を機械的に検出)。
- **I2Cキャプチャ既定 1s → 6s** + **conn空推奨** で偽failを構造的に削減。
- **Wokwiの部品スクショ**で「シム上のOLED表示」をAI単独で閉ループ化(今回の成功の決め手)。

## 4. 残るネック(分類・重大度・可動性)

| # | ネック | 種別 | 重大度 | 可動性 |
|---|---|---|---|---|
| N1 | 実機OLED/LED/基板の実ピクセル・物理状態が見えない | 構造(B) | 高 | 要ハード(USBカメラ → `evidence/latest/` に画像)。機材前提 |
| N2 | Wokwiキャンバスの座標が取れず配線レイアウトが手探り | 構造寄り(B) | 中 | `wokwi-cli mcp`(実験的)が突破口候補。当面は人間スクショ |
| N3 | シムと実機の乖離(チャージポンプ等タイミング/電源/アナログ) | 構造 | 中 | 不可避。実機検証で補うが N1 へ連鎖 |
| N4 | ハード操作のゲート往復(毎回人間承認) | 意図的(A) | 中 | 安全とのトレードオフ。「セッション単位の許可枠」で緩和可だが安全モデルが薄まる |
| N5 | MCP未実装 → AIは Bash で scripts を叩く | 付帯 | 低〜中 | 設計あり(`MCP_SETUP.md`)。表に `capture_uart`/`probe`/論理分割系の網羅漏れ |
| N6 | picotool 不在(reboot/info/UF2ロード等が openocd 頼み) | 付帯 | 低 | `brew install picotool` で解消可 |
| N7 | 証拠スキーマに視覚証拠(スクショ)の枠がない | 付帯 | 低 | Wokwiスクショや将来のカメラ画像を `*_result.json` に紐付ける枠を足せる |

## 5. 核心: 構造的天井 vs 付帯的摩擦

- **付帯的摩擦(削れる)**: N5/N6/N7、キャプチャ既定値、conn、診断ツール。本セッションでかなり削った。
  Labの「再帰的エンジニアリング」でAI自身が削り続けられる領域。**ここは順調。**
- **構造的天井(削りにくい)**: **N1(物理観測)と N3(シム乖離)。**
  Labの思想「物理→テキスト化」はデジタル計測(UART/ロジアナ/GDB)では成功しているが、
  **「パネルが光ったか」「線が挿さったか」という最後の一寸はテキスト化されていない。**
  N3で実機検証が必要になり、N3 → N1 へ連鎖して人間依存に着地する。
  **本セッションの往復の本質はこの連鎖。**

## 6. 次に効くレバー(順位)

1. **N1: 実機の「目」を1つ足す。** USBカメラ等で `evidence/latest/oled_photo.png` を撮る入口を作れば、
   Wokwi部品スクショの「実機版」が成立し、最大ネックが大きく緩む(要機材)。
   今回の「文字は出ているか」往復がゼロになる。
2. **N2: `wokwi-cli mcp` を調査。** Wokwi公式の実験的MCPが diagram/部品情報やシム操作を露出するなら、
   キャンバス盲目とWokwi操作の両方に効く。
3. **N5: MCP薄ラッパーの最小実装 + 提供ツール表の網羅更新。**
   Bash経由の現状でも動くが、操作面を固定し I/O を標準化すれば安全と再現性が上がる。
4. **N7: 視覚証拠を証拠スキーマに正式化**(Wokwiスクショ → 将来カメラ)。

## 7. 結論

- **ソフト/シム/CI/判定/自己改善のループはすでに「ほぼ完全自律」。**
  AIが環境自体を直せることも本セッションで実証済みで、ここがこのLabの最大の強み。
- **残る本質的ネックは2つだけ**: (1) 実機の物理観測(特に表示パネルと配線/電源の状態)を
  AIが直接見られないこと、(2) シムと実機の乖離がそれを呼び込むこと。
  本セッションの往復はすべてこの2つに帰着する。
- 付帯的な摩擦(ツール/既定値/診断/ドキュメント)は本セッションで実質的に削れた。
  **次に投資価値が高いのは「実機の目」(N1)** で、ここを埋めると人間依存が質的に下がる。
