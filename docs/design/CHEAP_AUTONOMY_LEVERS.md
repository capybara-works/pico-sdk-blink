# 安価で確実な自律性レバー (Cheap Autonomy Levers — Design Memo)

- **区分**: 設計メモ (Living)
- **背景**: [../reports/AGENT_AUTONOMY_ASSESSMENT.md](../reports/AGENT_AUTONOMY_ASSESSMENT.md) で、残る本質的ネックは
  **N1(実機の物理観測)** と **N3(シムと実機の乖離)** に集約されると結論した。
  本書は「**安価かつ確実に**そのネックを削る手段」を網羅的に洗い出し、優先順位を付ける。

## 中核原則: 「Pico自身を計測器にする」

このLabの思想は「物理→テキスト化」。最も安価で確実な実装は、**新しい機材ではなく
ファームウェア**である。RP2040は外付け部品ゼロで以下を自己計測でき、結果は既存の
UART証拠経路(`scripts/capture_uart.sh` → `uart_result.json`、`wait-serial` で判定)に
そのまま乗る。確率的なアナログ観測ではなく**決定論的なデジタル値**なので信頼性も高い。

- **VSYS(電源電圧)**: ADC3(GPIO29)が VSYS/3 を測る。給電の有無・概況が数値で分かる。
- **内蔵温度**: ADC4。
- **VBUS有無**: GPIO24(デジタル)。USB給電されているかを判定。
- **I2Cデバイス検出**: 既存のスキャン(0x3C 等)。
- **書き込んだ内容のCRC**: OLEDフレームバッファ等の「何を書いたか」を要約して報告。
- **自由ヒープ/ビルドID**: 異常・取り違えの検出。

→ これらを起動時に1ブロックで吐く **Power-On Self-Test(POST)** にすれば、AIは毎ブート
ごとに「実機の状態スナップショット」をテキストで得られる。**コスト$0、高信頼。**

## レバー一覧(安価・確実な順)

| # | レバー | 概算費用 | 信頼性 | 攻めるネック | 工数 |
|---|---|---|---|---|---|
| L1 | **ファームPOST(UART自己申告)** | **$0** | 高(決定論) | N1電源/生存/書込・N4 | 小 |
| L2 | **GPIOループバック自己診断** | ~$0(ジャンパ1本) | 高 | N1配線/ピン機能 | 小 |
| L3 | INA260 電源キャプチャ(導入予定) | ~$10 | 高 | N1電源(精密) | 小 |
| L4 | **安価USBウェブカメラ → 画像証拠** | ~$10–20 | 中(粗い判定) | N1表示ピクセル | 中 |
| L5 | reset_target.sh + セッション単位許可 | $0(運用) | — | N4往復 | 小 |
| L6 | はんだ付け/既製ハーネス | ~$5 | 高 | 接触不良の再発 | 小 |

### L1. ファーム Power-On Self-Test(最優先・$0) — ✅ 実装済み(2026-06-16)
起動時に構造化された自己診断行をUARTへ出力する。実装済みの形式:
```
POST fw=blink-i2c-oled vsys_mv=<int> temp_mc=<int> vbus=<0|1> i2c_oled=<0|1>
OLED updated fbcrc=0x<16bit>   # 描画ごとに「何を書いたか」の指紋
```
- 計測元(外付け部品ゼロ): `vsys_mv`=ADC3(GP29=VSYS/3)、`temp_mc`=ADC4内蔵温度、
  `vbus`=GP24センス、`i2c_oled`=0x3C検出、`fbcrc`=フレームバッファCRC16。全て整数(float printf回避)。
- **読み取り**: 既存 `capture_uart.sh`。`blink.test.yaml` / `blink_i2c.test.yaml` に
  `wait-serial: 'POST '` を追加済み(CI/Wokwiでも行の存在を判定)。実機リブートは `scripts/reset_target.sh`。
- **効果**: 今回最大の躓き=ファントム給電を、**`vsys_mv`が低い/`i2c_oled=1`なのに描画系が異常**として
  AIが数値で切り分け可能に。人間の目に頼っていた「給電されてる?」「ちゃんと起動した?」が消える。
- **限界(正直)**: `fbcrc` は「何を書いたか」を保証するが**画素が光ったかは保証しない**(SSD1306はI2Cで
  確実な読み戻しが難しい)。点灯の確証はL3(電流)とL4(画像)で補う。
  **Wokwiは電源/ADCを模擬しない**ため `vsys_mv`/`temp_mc`/`vbus` は非物理値で、値が意味を持つのは実機。
- **実機検証(2026-06-16)**: フラッシュ→`reset_target.sh`→`capture_uart.sh` で実値取得。
  `temp_mc`(29.3°C)・`vbus=1`・`i2c_oled=1` は実機で**正しいと確認**。一方 `vsys_mv` は
  ~1.7–1.85Vと実機で**異常**(捨て読みを入れても不変)= **この基板のGP29にVSYS/3分圧が無い**
  (安価なRP2040クローンに多い)。POST機構自体は正常で、3/4フィールドが検証済み。
  **精密な電源測定はINA260(L3)に委ねる**のが正しい切り分け。実機がシムで出ないバグを露呈した好例。

### L2. GPIOループバック自己診断(~$0)
余ったGPIOを2本ジャンパで繋ぎ、片方を駆動・他方で読む。一致をUART報告。
- **効果**: ロジアナや人間の目なしで、**ピンの導通・機能・取り違えを電気的に確定**。
  今回のSCLタップ接触不良の類を、配線側でも自己検出できる。
- **限界**: ループした2本のみ検証。GPIO2本+ジャンパを消費。

### L3. INA260 電源キャプチャ(導入予定・~$10)
I2Cで電圧/電流/消費電力を精密取得 → `scripts/capture_power.sh` → `power_result.json`。
詳細・位置づけは別途([../reports/AGENT_AUTONOMY_ASSESSMENT.md](../reports/AGENT_AUTONOMY_ASSESSMENT.md) の N1電源軸)。
- **効果**: 「0x3C ACKだが電流≒0mA → 無給電/無負荷」を**数値で即断**。POST(L1)の電源推定を実測で裏取り。
- **要確認**: WokwiにINA260部品があるか(INA219系はあるが260は未確認)。無ければ実機専用ステップ。

### L4. 安価USBウェブカメラ → 画像証拠(~$10–20)
OLEDに固定したウェブカメラで撮影 → `scripts/capture_photo.sh` → `evidence/latest/oled_photo.png`。
- **要点**: **AIはこの画像を(Wokwi部品スクショと同様に)そのまま視覚的に読める**ので、OCRは不要。
  「点灯しているか/文字が出ているか」程度なら十分確実。今回の「文字出てる?」往復がゼロになる。
- **限界**: 画素単位の正確性や微細な描画崩れの判定は不確実。固定マウントと照明の安定が前提。
  カメラ自体は安価だが、**治具・自動化を作り込むと費用と工数が増える**(そこから先が資金フェーズ)。

### L5. reset_target.sh + セッション単位の許可(運用・$0)
- `scripts/reset_target.sh`(openocd reset)でリフラッシュせずPOSTを再実行できるようにする。
- ハード操作の毎回承認(N4)を、**人間が時間/回数を区切って一括許可**する枠を設ける案。
  往復を減らせるが**安全モデルが薄まる**ため、opt-in・ログ必須・短時間限定とする(トレードオフを明示)。

### L6. はんだ付け/既製ハーネス(~$5)
Dupont線の接触不良(今回のloose-SCL)を物理的に潰す。地味だが確実。L2が検出、L6が予防。

## 安価では届かない領域(正直な線引き)

以下は**確実にやるなら資金が必要**で、本書の「安価レバー」では代替しない:
- **画素単位の表示正当性**(粗い点灯判定を超える): 高解像度カメラ＋治具＋画像処理。
- **アナログ波形・ノイズ・タイミング余裕**: 実オシロスコープ。
- **物理的な再配線・抜き挿しの自動化**: ロボティクス。

## 推奨する着手順(安価・確実ファースト)

1. **L1 ファームPOST**(今すぐ・$0)— 最大の費用対効果。実機状態の常時テキスト化。
2. **L3 INA260**(導入予定)— L1の電源推定を実測で裏取り。
3. **L4 ウェブカメラ**(~$10–20)— L1/L3が触れない「画素が出たか」を、AIが直接読む画像で確定。
4. **L2 / L5 / L6** — 必要に応じて(配線信頼性・往復削減)。

L1→L3→L4 を揃えると、今回人間に依存した観測(給電・起動・I2C・電流・表示)の**大半がテキスト/画像の
証拠に置き換わり**、N1の実用的な大部分を**安価に**埋められる。

## 統合上の注意(既存規約)

- 新しい証拠ステップ(`power`/`photo`等)を足す場合は **4箇所同期**が必要:
  `summarize_evidence.py` の `STEPS`、`common.sh` の `reset_evidence_dir`、`verify_all.sh`、
  `ci_phase1_smoke.sh` の許容ステータスmap。
- **POST(L1)は新ステップ不要**(既存UART経路に乗る)。最小工数で入る。
- ゲートは既存方針を踏襲: 電源/カメラ実測は `PICO_HARDWARE=1`(必要なら `PICO_POWER`/`PICO_CAMERA` の
  個別ゲット)。AIが自己有効化しない原則は維持。
- 画像証拠は将来的に証拠スキーマへ「視覚証拠」枠として正式化(アセスメントN7)。
